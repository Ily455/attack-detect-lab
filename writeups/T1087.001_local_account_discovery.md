# T1087.001 — Account Discovery: Local Account

## Technique

**MITRE ATT&CK:** [T1087.001](https://attack.mitre.org/techniques/T1087/001/)  
**Tactic:** Discovery  
**Platform:** Linux

Adversaries enumerate local accounts to understand what users exist on a system, identify privileged accounts, and find targets for lateral movement or privilege escalation. On Linux, this is typically done through standard system commands that require no special tools.

---

## Test Environment

- **Target:** Ubuntu 22.04 ARM64 (UTM VM)
- **Attacker:** macOS M3 triggering via SSH
- **Tool:** Atomic Red Team — `Invoke-AtomicTest T1087.001`
- **Logging:** auditd with execve syscall rules, shipped via Filebeat to Elasticsearch

---

## Execution

```bash
./run-test.sh T1087.001
```

Atomic ran enumeration commands including: `cat /etc/passwd`, `cat /etc/group`, `cat /etc/shadow`, `lsof`, `who`, `last`, `getent passwd`.

---

## Evidence

auditd captured every enumeration command via `execve`. Examples:

```
type=EXECVE msg=audit(...): argc=2 a0="cat" a1="/etc/passwd"
type=EXECVE msg=audit(...): argc=1 a0="lsof"
type=EXECVE msg=audit(...): argc=2 a0="getent" a1="passwd"
```

The `lsof` output in particular produces a large volume of log data — every open file descriptor for every process on the system. In a real investigation, this volume itself is a signal: a process generating hundreds of read events against system files in quick succession is worth investigating.

---

## Detection

### KQL (Kibana)

```
message: "type=EXECVE" AND (message: "a1=\"/etc/passwd\"" OR message: "a1=\"/etc/shadow\"" OR message: "a1=\"/etc/group\"")
```

Or targeting enumeration tools:

```
message: "type=EXECVE" AND (message: "a0=\"lsof\"" OR message: "a0=\"getent\"")
```

See [`detections/T1087.001_local_account_discovery.yml`](../detections/T1087.001_local_account_discovery.yml)

## Notes

- Discovery techniques like this generate low-severity signals individually — they matter most in context, when combined with other techniques from the same session
- `cat /etc/shadow` requires root — if this appears in logs from a non-root process, it's a strong anomaly
- `lsof` run as root producing full system-wide output is unusual outside of active debugging sessions
