# T1136.001 — Create Account: Local Account

## Technique

**MITRE ATT&CK:** [T1136.001](https://attack.mitre.org/techniques/T1136/001/)  
**Tactic:** Persistence  
**Platform:** Linux

Adversaries create local accounts to maintain access to a compromised system. A new account provides a persistence mechanism that survives credential rotation on existing accounts and may go unnoticed if account auditing is not in place.

---

## Test Environment

- **Target:** Ubuntu 22.04 ARM64 (UTM VM)
- **Attacker:** macOS M3 triggering via SSH
- **Tool:** Atomic Red Team — `Invoke-AtomicTest T1136.001`
- **Logging:** auditd with execve syscall rules, shipped via Filebeat to Elasticsearch

---

## Execution

```bash
./run-test.sh T1136.001
```

Atomic ran 5 sub-tests. 2 succeeded on Linux:

1. **T1136.001-1** — Create a standard user account (`evil_user`)
2. **T1136.001-6** — Create a user with UID=0 and GID=0 — effectively a second root account

3 tests failed due to platform mismatch (FreeBSD `pw` command not available) or missing tooling (`kubectl`).

---

## Evidence

auditd captured both the account creation and deletion. The cleanup PROCTITLE record (hex-encoded):

```
type=PROCTITLE proctitle=7573657264656C006576696C5F75736572
```

Decoded: `userdel evil_user`

The SYSCALL record confirms the binary used and the calling context:

```
type=SYSCALL exe="/usr/sbin/userdel" auid=1000 uid=0 AUID="ubuntu" UID="root"
```

`auid=1000` is the audit UID — the original logged-in user (`ubuntu`) before privilege escalation. This field persists across `sudo` and is key for attribution: even though the command ran as root, auditd knows it was initiated by `ubuntu`.

---

## Detection

### KQL (Kibana)

```
message: "type=EXECVE" AND (message: "a0=\"useradd\"" OR message: "a0=\"adduser\"")
```

## Notes

- `auid` (audit UID) is one of the most useful fields in auditd for attribution — it survives sudo and tracks the original user session
- Test 6 (UID=0 account) is particularly notable — a user with UID=0 has full root access regardless of the username. This is harder to detect casually since `id` and `whoami` return "root" for any UID=0 user
- Account creation by a non-system process at an unusual time is a reliable signal — legitimate provisioning typically happens during business hours via known automation
- Check `/etc/passwd` modifications alongside `useradd` events for confirmation
