# T1053.003 — Scheduled Task/Job: Cron

## Technique

**MITRE ATT&CK:** [T1053.003](https://attack.mitre.org/techniques/T1053/003/)  
**Tactic:** Persistence  
**Platform:** Linux

Adversaries abuse cron to schedule malicious commands that execute automatically at set intervals. Common persistence vectors include writing to `/etc/cron.d/`, `/var/spool/cron/crontabs/`, or modifying per-user crontabs via `crontab -e`. Because cron runs as root by default, this technique often provides privileged persistence.

- **Target:** Ubuntu 22.04 ARM64 (UTM VM)
- **Attacker:** macOS M3 triggering via SSH
- **Tool:** Atomic Red Team — `Invoke-AtomicTest T1053.003`
- **Logging:** auditd with execve syscall rules, shipped via Filebeat to Elasticsearch

## Execution

```bash
./run-test.sh T1053.003
```

Atomic ran 4 sub-tests:
1. Replace crontab with referenced file
2. Add script to all cron subfolders (`/etc/cron.daily`, `/etc/cron.hourly`, etc.)
3. Add script to `/etc/cron.d/`
4. Add script to `/var/spool/cron/crontabs/` — file named `persistevil`

**Result:** 4/4 tests succeeded.

---

## Evidence

auditd captured the `crontab` binary being invoked with a file from `/tmp/` as argument — the clearest signal from this technique:

```
type=EXECVE msg=audit(1780492332.445:1809): argc=2 a0="crontab" a1="/tmp/persistevil"
```

`crontab` loaded from `/tmp/` is anomalous — legitimate crontab management doesn't source files from `/tmp/`. The companion PATH record confirms the binary:

```
type=PATH msg=audit(1780492332.445:1809): item=0 name="/usr/bin/crontab" nametype=NORMAL
```

The PROCTITLE record from the cleanup phase (hex-encoded command line) was also captured:

```
type=PROCTITLE proctitle=62617368002D6300726D202F7661722F73706F6F6C2F63726F6E2F63726F6E746162732F706572736973746576696C202D66
```

Decoded: `bash -c rm /var/spool/cron/crontabs/persistevil -f`

---

## Detection

### KQL (Kibana)

```
message: "type=EXECVE" AND message: "a0=\"crontab\""
```

High-confidence variant — crontab loaded from /tmp specifically:

```
message: "type=EXECVE" AND message: "a0=\"crontab\"" AND message: "/tmp/"
```

## Notes

- Writing to cron directories requires root — this technique is a strong indicator of privilege escalation having already occurred
- The `PROCTITLE` field in auditd records stores the full command line as a hex string when it contains special characters — decode it to get the actual command
- Legitimate cron management by package managers generates the same events — correlation with the creating process (`ppid`, `comm`) is needed to distinguish
